# 014: 客户端角色分割 — Session State 模式切换

- **日期**: 2026-07-09
- **状态**: 已实施

## 背景

当前 Streamlit 在同一界面中同时暴露"数据导入（上传/URL 抓取）"和"市民咨询（问答对话）"两类操作。用户不分角色，所有功能平铺可见，界面定位模糊。

这个问题的本质是**产品缺少角色区分**：数据维护者（管理员）和信息消费者（市民）的使用场景完全不同。

## 生产级做法 vs 项目当前阶段

### 生产级（长期目标）

在前后端分离的架构下：

```
前端                              后端
admin.example.com  ─── API ──→  用户系统 (users表 + JWT + 角色中间件)
app.example.com    ─── API ──→    ├── 管理员: 文档管理、知识库维护
                                  └── 市民: 仅问答对话
```

- 前后端分别独立部署，不同域名或子路径
- 用户表含 `role` 字段（admin / staff / citizen）
- JWT 认证 + 路由中间件鉴权
- 前端根据角色渲染不同页面和导航

这是成熟产品的标准做法，但前提是有独立的前端应用和用户基础设施。

### 项目当前阶段

项目仍处于 Streamlit 单页面 + 无用户系统的阶段。如果在这个阶段强行做数据库用户表、登录注册、角色鉴权，代价是：

1. **Streamlit 本身不支持多页面路由** — 登录态管理需全部手写在 session_state
2. **引入用户系统至少增加一周工作量** — 偏离当前核心目标（RAG + Agent）
3. **接口层改动大** — 所有路由加认证中间件，与现有瘦客户端架构耦合

## 决策

**在当前 Streamlit 下使用 Session State 模式切换实现角色分割，设计上为未来真正的前后端分离预留扩展点。**

### 方案：Session State 模式切换

```python
# 当前方案（待实施）
session_state.mode
    ├── "user"   → 市民咨询界面（默认，仅对话）
    └── "admin"  → 管理后台界面（导入/上传/知识库管理）
```

#### 实现要点

| 维度 | 做法 |
|------|------|
| 切换方式 | 侧边栏顶部两个按钮 Tab |
| 状态管理 | `st.session_state.mode` 持久化 |
| 界面切换 | 纯条件渲染，不重新运行脚本 |
| 管理面板安全 | 本阶段无密码保护（Streamlit 本地开发环境） |
| 角色切换时 | **清空对话历史**（`st.session_state.messages = []`） |

**清空历史的理由：** 当前对话未持久化，仅存在于内存中。未来持久化后，用户对话会存入数据库按 `user_id` 查询，不存在"切管理员丢市民数据"的问题。现在不清空反而会导致角色切换后界面混乱。清空是合理且无损失的选择。

#### 保留的扩展点

```
session_state.mode
  ↓（未来替换为）
从 JWT token 解析 role 字段 → 决定渲染哪个视图
```

当前代码中所有根据 `mode` 判断视图的逻辑，未来只需将 `mode` 的来源从 Session State 按钮切换，替换为登录后的角色信息即可。**视图渲染逻辑本身无需大改。**

### 不做的范围

- ❌ 不做用户系统（数据库、注册、登录）
- ❌ 不做前后端分离（仍为单一 Streamlit 进程）
- ❌ 不做 API 鉴权（所有路由保持无状态）
- ❌ 不做管理面板密码保护（本地开发阶段不需要）
- ❌ 不修改现有测试（94 个测试全部保持通过）
