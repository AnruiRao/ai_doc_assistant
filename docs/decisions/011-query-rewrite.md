# 011: Query Rewrite — LLM 查询改写（V3 实验 A1）

- **日期**: 2026-06-28
- **状态**: 已废弃 — 详见下方"V3 阶段全量评测"

## 背景

V3 收敛时，列举型 query（"有哪些"、"包含哪些"）Faithfulness 持续低迷：

| Query | V3 F |
|-------|------|
| #12 VectorStore 核心方法 | 0.289 |
| #14 RAG Tool save/search | 0.393 |
| #19 异常体系可重试类 | 0.194 |

根因：答案散落在 3-5 个不连续的 chunk 中，单次向量检索无法完整召回。

## 选项对比

| 方案 | 原理 | 新依赖 | 复杂度 | 预期效果 |
|------|------|--------|--------|---------|
| **Query Rewrite（LLM 改写）** | 识别列举/对比/多维度型问题，拆成子查询，多路检索合并去重 | 无 | ★★ | 列举型 F 显著提升 |
| 扩大 top-k | 粗暴增加召回量 | 无 | ★ | 噪音同比例放大，精度不升反降 |
| Reranker | 粗召回 → 精排 | sentence-transformers | ★★ | 提升精度但不解决覆盖面问题 |
| HyDE | 生成假答案再用假答案检索 | 无 | ★★ | 多一次 LLM 调用，query-chunk gap 大的场景有效 |

## 决策

**V3 实验 A1 选择 Query Rewrite**，理由：

1. **零新依赖**：纯 LLM prompt，不引入新库
2. **直击根因**：列举型需要多路查询才能覆盖全部答案片段，扩大 top-k 或 Reranker 都治不了这个根因
3. **实现内聚**：逻辑封装在 `RagTool.search_raw()` 内部，Agent 和评测脚本零感知
4. **消融友好**：通过 `enable_query_rewrite` 开关控制，可独立评测
5. **后续可组合**：QR 解决覆盖面问题后，Reranker 可在此基础上提升精度，两者互补

### 放弃的理由

- **扩大 top-k**：只是把问题推迟到更靠后的位置，不是真正的解法
- **HyDE**：多一次 LLM 调用增加延迟，且当前 query-chunk 语义 gap 不大（R 值已 0.88+），ROI 不高

## 架构改动

### 新增文件

`src/retrieval/query_rewriter.py` — LLM 驱动的查询改写：

```python
class QueryRewriter:
    def rewrite(self, query: str) -> list[str]:
        # prompt 识别列举/对比/多维度 → 拆分子查询
        # 简单事实型 → 返回 [原query]
        # JSON 解析失败 → 兜底返回 [原query]
```

### 修改文件

| 文件 | 改动 |
|------|------|
| `src/core/config.py` | 加 `enable_query_rewrite: bool` 开关 |
| `src/tools/impl/rag_tool.py` | 抽 `search_raw()` 方法：rewrite → 多路检索 → `(source, chunk_index)` 去重合并 |
| `scripts/evaluate_rag.py` | contexts 收集改为调 `rag_tool.search_raw()`，与 Agent 内部检索统一入口 |

### 关键设计决策

1. **同步方法**：`rewrite()` 和 `search_raw()` 都是同步的，运行在 `asyncio.to_thread` 的线程内，无需 async
2. **懒初始化**：QueryRewriter 在 `RagTool` 首次调用 `_get_rewrite()` 时才创建，减少空跑开销
3. **兜底策略**：LLM 输出解析失败 → 返回 `[原query]`，不打断 Agent 流程
4. **去重 key**：`(source, chunk_index)`，同一 chunk 被不同子查询命中时只保留一份
5. **评测入口统一**：`search_raw()` 同时被 `run()` 和评测脚本调用，后续加 Reranker 只改一处

### 退化分析

退化原因归纳为三类：

**1. LLM 分类器不准** — 应不拆的问题被误判为多维度型并拆分
- `文档上传到检索步骤`（流程型 → 误拆 → 碎片化）
- `向量库目录`（事实型 → 误拆 → 噪声）

**2. 子问题碎片化** — 列举型拆分后，k=4 结果被平分到每个子问题，深度减半
- `RAG Tool save/search`：拆为"sav e做了什么"+"search做了什么"，各自只检索到单方面，丢失了"两者对比"的整合视角
- `异常体系可重试类`：拆开后各子问题只命中部分信息

**3. 子问题完全替代原问题** — 当前实现用子问题替换原问题检索，而非补充
- 即使正确拆分，子问题也只覆盖到局部，原问题的全局视角丢失

### 关键发现

QR 净提升 **仅 +0.017**（0.7028 → 0.7202），几乎在误差范围内。但带来了明显的退化 case：
- 受益类（+0.385 ~ +0.146）：多维度/列举型，扩大召回面确实有效
- 受害类（-0.250 ~ -0.124）：流程型/简单事实型，被误拆或碎片化

### 结论

**V3 阶段弃用 Query Rewrite**，理由：

1. **净收益太低**：+0.017 F，不足以抵消退化 case 的负面影响
2. **分类器不稳定**：LLM 作为分类器在小规模 20 条 query 上就频繁误判，规模放大后更不可靠
3. **架构有结构性缺陷**：子问题替代而非补充原问题，一旦误拆无法自救
4. **Reranker 是更好的方向**：它不改变召回面、对所有问题类型一致受益、没有"拆碎了"的副作用
